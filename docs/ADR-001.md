# ADR-001: CA / VNS 平级并行架构

## 状态

**已替换（2026-06）**——原串行方案已废弃。

**之前是什么：** SA + VNS 串行流水线。SA 全参数搜索（遍历 6 种聚类方法 × k 值）找到最优分组，再将每组初始解传递给 VNS 精炼。

**为什么废弃：** 实验数据（n60w60.001）表明 SA(k=1)=955.4 而 VNS(k=1)=620.3（理论最优），SA 的评分不能代表 VNS 的潜力。串行"SA 选参数→VNS 精炼"本质上是让低精度算法替高精度算法做决策，埋没了 VNS 的最优性。改为 CA / VNS 平级并行，两者互不依赖。

---

> 以下为新架构的完整设计说明，对应 `src/engine/` 目录。

## 一、设计原则

### 双引擎平级并行

CA（压缩退火）和 VNS（变邻域搜索）是**平级选项**，而非串行流水线。

| 引擎 | 文件 | 定位 | 精度 | 速度 |
|------|------|------|------|------|
| **CASolver** | `ca.py` | 快速预览 | 距最优 6-10%（与 GA/ACO 同级） | 秒级 |
| **VNSSolver** | `vns.py` | 深度优化 | 可达理论最优 | 分钟级 |

CA 的精度上限决定了它不适合替 VNS 做决策。CA 的价值在于快——秒级出方案让用户预览行程结构；VNS 的价值在于准——追求极致精度。两者面向不同场景，没有上下级关系。

### n_days 状态闸门

天数参数是整个流程的状态闸门：

```
未指定 n_days ──→ 只能走 ca_suggest()（CA 快速建议）
指定 n_days   ──→ 可走 cluster_and_solve(mode="fast"/"deep")
  mode="deep" + 无 n_days ──→ 直接报错
```

## 二、双阶段流程

```
用户请求
  │
  ├── 指定了 n_days ──→ cluster_and_solve(mode, n_days=N)
  │                        │
  │                        ├── 6 种聚类方法逐一分组
  │                        ├── mode="fast" → CA 求解各组
  │                        ├── mode="deep" → VNS 求解各组
  │                        └── 返回最优方案
  │
  └── 未指定 n_days ──→ ca_suggest()
                           │
                           ├── 全参数搜索（min_days~max_days × 6 聚类方法）
                           ├── 增益阈值式早退
                           ├── 去重（frozenset 分组集合去重）
                           ├── 按成本排序取 top-5
                           └── 返回建议，等待用户指定 n_days
```

## 四、关键设计决策

### 4.1 早退机制

从"连续无改善计数"改为**增益阈值式**：

- 改善 $\ge$ `early_stop_gain_threshold`（默认 1.0%）→ 有意义，重置计数器
- 改善 < 1.0% → 算无效，`worse_count += 1`
- 连续 3 次无效 → 停止该聚类方法的搜索
- 参数托管在 `CA_DEFAULT_PARAMS` 中

### 4.2 去重逻辑

`_deduplicate()` 基于 `frozenset(frozenset(g) for g in groups)` 去重。

**先去重后排序 → 先排序后去重**：不同聚类方法可能产出相同分组方案，但 CA 有随机性，靠后运行的方法成本可能更低。因此改为**先按 cost 排序**，再**去重保留第一个**（即成本最低的），确保每个分组下保留最优解，而非方法注册表顺序优先。

### 4.3 命名规范

| 旧命名 | 新命名 | 说明 |
|--------|--------|------|
| SASolver | CASolver | 正名为压缩退火，不是标准 SA |
| k | n_days | 天数语义，不叫"聚类数" |
| min_clusters / max_clusters | min_days / max_days | 对应天数范围 |
| best_k | best_days | 最优天数 |

### 4.4 接口变更工作流

修改公开 API 时：

1. 先更新 `src/engine/__init__.py`（或 `src/data/__init__.py`）顶部的接口清单注释块
2. 再修改对应的实现文件
3. 确保接口清单与 `__all__` 列表保持同步

## 五、模块职责

| 文件 | 职责 |
|------|------|
| `ca.py` | CASolver：压缩退火主循环 + 标准/时间窗引导邻域 + 2-opt |
| `vns.py` | VNSSolver：Shake → VND → SA 接收 + 精英池 + 自适应算子权重 |
| `clustering.py` | 6 种聚类方法注册表 + 统一调用入口 |
| `fitness.py` | 适应度计算：距离 + 等待惩罚 + 迟到惩罚 + 晚归惩罚 |
| `search.py` | ca_suggest() + cluster_and_solve() + solve_groups() + _deduplicate() |
| `pipeline.py` | run_planning()：双阶段流程编排 + Cesium 可视化生成 |

## 六、数据流

```
run_planning(poi_cache, mode, n_days)
  │
  ├── build_real_data()                   高德 API → 成本矩阵
  │
  ├── cluster_and_solve(spots, depot, dist_mat, mode, n_days)
  │     │
  │     ├── ca_suggest()                  未指定天数 → 返回 top-5 建议
  │     │     ├── 遍历聚类方法 × n_days
  │     │     ├── 增益阈值早退
  │     │     ├── 去重 + 排序取 top-5
  │     │     └── 返回 {"type": "suggestion"}
  │     │
  │     └── 指定天数
  │           ├── 遍历 6 种聚类方法
  │           ├── solve_groups(solver_type="CA"/"VNS")
  │           └── 返回 {"type": "solution"}
  │
  │
  ├── type="suggestion" → 返回给用户，等待指定 n_days
  │
  └── type="solution"
        ├── 生成每日行程（时间窗收缩 + 到达状态判断）
        ├── generate_cesium_html()         3D 路线可视化
        └── 返回完整结果
```
