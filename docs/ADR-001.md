# ADR-001: SA + VNS 串行 → CA / VNS 平级并行

## 状态

已替换（2026-06）——原串行方案已废弃，见下方说明。

## 背景

原项目（TSPTW visualization 分支）使用 VNS+ 单一求解器。TravelPal 初始设计采用 SA + VNS 串行方案（ADR-001 初版），即 SA 全参数搜索→VNS 精炼的流水线。

实际实验数据暴露了串行方案的根本问题：SA 的评价标准完全不能代表 VNS 的潜力，串行只会埋没 VNS 的最优性。

## 关键实验证据

基于 `n60w60.001` 数据集：

| 方案 | 总成本 | 说明 |
|------|--------|------|
| 理论最优 | 620.3 | — |
| VNS(k=1) | 620.3 | 达理论最优 |
| SA(k=1) | 955.4 | 被串行错误选中的"最优" |
| SA(k=3) | 680.0 | 被串行正确选中，但精度仍不如 VNS |

- **SA(k=1) 和 SA(k=3) 的差距巨大**，而串行方案依赖 SA 的评分来选择和参数，当 SA 找到 k=1 时，后续 VNS 连出场机会都没有就被淘汰了。
- **VNS(k=1) 却能达到理论最优**，说明 SA 的评分和 VNS 的潜力之间没有正相关关系。

### 根本原因

SA（包括压缩退火 CA）的精度上限约为距最优 6-10%，与原生 GA、ACO 同级。而 VNS 可以达到理论最优级别。串行"SA 选参数→VNS 精炼"本质上是让一个低精度算法替高精度算法做决策，这在数学上就不成立。

## 方案对比

| 方案 | 精度 | 运行时间 | 用户可交互 | 代码复杂度 |
|------|------|---------|-----------|-----------|
| **串行**（SA 选参→VNS 精炼） | 被 SA 选参拖累 | 中 | 可在中间介入 | 高 |
| **平级并行**（选定方案） | VNS 不受 SA 约束 | 低（fast）/ 高（deep） | 用户/Agent 按需选择 | 低 |
| **VNS 单一** | 最高 | 高 | 无快速方案 | 低 |

## 选定方案：CA / VNS 平级并行

### 架构

```
用户输入景点列表
      ↓
cluster_and_solve(spots, depot, dist_mat, mode, n_days)
      ↓
  ┌─── mode="fast" ────┐     ┌─── mode="deep" ────┐
  │  CA（秒级方案）     │     │  VNS（分钟级方案）  │
  │  - 精度距最优 6-10% │     │  - 可达理论最优     │
  │  - 适合快速预览     │     │  - 适合最终定稿     │
  │  - 完全自包含       │     │  - 完全自包含       │
  └────────────────────┘     └────────────────────┘
```

### n_days 参数

- **用户指定 n_days 时**：k 固定为该值，无需任何搜索，跳过聚类方法遍历。
- **用户未指定 n_days 时**：按节点规模自动估算 k 范围，用简单聚类（而非全参数搜索）确定分组。
- 旧方案中 SA 枚举 6 种聚类方法 × k 值来搜索分组，现已完全移除。SA 从不具备替 VNS 决策的能力，不应再承担此角色。

### 核心变更

| 项目 | 旧方案 | 新方案 |
|------|--------|--------|
| 命名 | SASolver | CASolver（压缩退火，不是标准 SA） |
| 协作关系 | SA 搜索结构 → VNS 精炼（串行） | CA / VNS 平级，按 mode 选择 |
| 聚类搜索 | SA 枚举 6 种方法 × k 值 | n_days 固定 / 节点规模估算 |
| 入口函数 | sa_vns_pipeline() | cluster_and_solve(..., mode=...) |
| 用户接口 | run_planning() 固定串行 | run_planning(..., mode=...) |
| 代码量 | 两套参数管理，中间状态传递 | 各自独立，互不依赖 |

### 为什么 CA 和 VNS 是平级的

CA 的精度上限决定了它不适合替 VNS 做任何决策。CA 的价值在于**快**——秒级出方案，让用户对行程结构有个快速预期。VNS 的价值在于**准**——追求极致精度。两者面向不同的使用场景，没有上下级关系。

## 后续考虑

- 多日行程默认走 fast 模式，用户确认后再切 deep 精炼
- 单日行程直接走 deep 模式（VNS 在单日小规模问题上快速收敛）
- Agent 可根据景点数量、用户意图（"快速看看" vs "确定行程"）自动选择 mode
- `CASolver` 这个正名会沿用到后续所有算法引用中，不再出现 SASolver
